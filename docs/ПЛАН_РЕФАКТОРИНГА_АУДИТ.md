# План рефакторинга по отчёту аудита архитектуры

План составлен по [ARCHITECTURE_AUDIT.md](ARCHITECTURE_AUDIT.md) (внутренний отчёт архитектора и вердикт внешнего аудитора). Задачи зарегистрированы в [ARTIFACTS.md](ARTIFACTS.md), спринт 9 — в [ПЛАН_РЕЛИЗА_1.3.md](ПЛАН_РЕЛИЗА_1.3.md).

**Важно:** набор задач спринта 9 перед реализацией будет существенно пересматриваться.

---

## 1. Цели рефакторинга (из аудита)

- Ввести/закрепить слой use case для ключевых сценариев (голосовой поиск, выбор клиники; сохранение протокола уже есть).
- Разгрузить крупные виджеты (ExaminationCreatePage, PatientsListPage): в виджетах — только UI и вызовы use case’ов.
- Унифицировать доступ к зависимостям в presentation: единое правило Riverpod/GetIt.
- Снизить связность навигации с доменом (redirect без прямого вызова репозитория).
- Устранить узкие места производительности (подсчёт пациентов через COUNT).
- Задокументировать правило DI для разработчиков.

## 5. Связь с правилами проекта

- **Структура (Clean Architecture):** рефакторинг сохраняет разделение data/domain/presentation; use case’ы и сервисы в domain (или application-слое), UI только вызывает их.
- **Артефакты:** все задачи имеют ID VET-NNN в [ARTIFACTS.md](ARTIFACTS.md); в коммитах указывать ID (например, `refactor(VET-181): use case голосового поиска`).
- **Тесты:** перед коммитом — `flutter analyze` без ошибок, `flutter test` — все тесты зелёные; новые сценарии покрыты по таблице выше.
- **Линтер:** соблюдать [analysis_options.yaml](../analysis_options.yaml), не отключать правила без явной просьбы.
- **Статусы:** не переводить задачи в «закрыто» автоматически; после реализации — «реализовано»/«выполнено», «закрыто» — только по подтверждению ответственного.
