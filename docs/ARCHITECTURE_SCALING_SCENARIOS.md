# Сценарии масштабирования Vet Assistant

Документ фиксирует три целевых сценария масштабирования и допущения по каждому. Используется для приоритизации архитектурных решений и плана внедрения.

---

## 1. Сценарий до 10 000 пользователей

**Допущения:**
- Офлайн-ориентированная модель: данные хранятся локально (Drift/SQLite) на устройстве.
- Одиночные инсталляции: одна клиника / один врач — одно приложение на устройстве (или несколько устройств без обязательной синхронизации между ними).
- Редкая синхронизация: экспорт/импорт (JSON/ZIP) для переноса данных между устройствами или бэкапа; централизованный бэкенд не обязателен.
- Лимиты бесплатной версии и объёмы данных остаются умеренными (десятки–сотни пациентов на инсталляцию).

**Временной горизонт:** текущий и ближайшие 1–2 релиза (рекомендуемый приоритет для стабилизации продукта).

**Критичные архитектурные требования:**
- Сохранить и усилить feature-based Clean Architecture, не вводя удалённый бэкенд.
- Выделить слой use case для ключевых сценариев; разгрузить крупные экраны от бизнес-логики.
- Унифицировать DI (GetIt + Riverpod) на выбранных экранах.
- Обеспечить отсутствие блокировки UI при тяжёлых операциях (PDF, экспорт/импорт).

---

## 2. Сценарий до 100 000 пользователей

**Допущения:**
- Появление централизованного бэкенда: пациенты, осмотры, шаблоны, профили/клиники могут храниться и синхронизироваться с сервером.
- Несколько устройств на одну клинику с синхронизацией данных.
- Базовые роли и разграничение доступа (врач, клиника, администратор).
- Офлайн-режим с очередью изменений и разрешением конфликтов при синхронизации.

**Временной горизонт:** среднесрочная перспектива (после стабилизации сценария 10k).

**Критичные архитектурные требования:**
- Абстракции репозиториев, допускающие реализации: локальная (Drift) и удалённая (HTTP/gRPC).
- Синхронизирующий слой (sync service) поверх локального и удалённого хранилищ.
- Контракты API, опирающиеся на существующие доменные модели.
- Аутентификация, многоклиничность, базовый аудит изменений на сервере.

---

## 3. Сценарий до 1 000 000 пользователей

**Допущения:**
- Многотенантный облачный сервис с высокой нагрузкой.
- Разделение чтения/записи (CQRS-подход при необходимости), резервирование, отказоустойчивость.
- Расширенный аудит, лицензирование, аналитика использования.
- Масштабируемая инфраструктура бэкенда (микро-/модульные сервисы или чётко модульный монолит).

**Временной горизонт:** долгосрочная перспектива.

**Критичные архитектурные требования:**
- Bounded context’ы на бэкенде: Пациенты/Осмотры, Шаблоны и Справочники, Профили/Клиники, Отчёты и PDF, Аудит и Лицензирование.
- Клиент: все интеграции с сервером только через интерфейсы репозиториев/сервисов; расширяемая модель синхронизации (очереди, версионирование, стратегии конфликтов).
- Логирование и трассировка для анализа поведения и инцидентов при росте базы пользователей.
- Конфигурируемые лимиты (пациенты, медиа, глубина истории) — не зашивать в код, выносить в конфигурацию.
